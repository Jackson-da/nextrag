## 3. 技术选型

### 3.1 技术栈总览

| 层级 | 技术选型 | 版本 | 说明 |
|------|---------|------|------|
| **前端框架** | Vue3 | 3.4+ | 组合式 API、性能优异 |
| **UI 组件库** | Element Plus | 2.5+ | 企业级组件、文档完善 |
| **构建工具** | Vite | 5.0+ | 快速启动、热更新快 |
| **后端框架** | FastAPI | 0.110+ | 高性能、自动文档 |
| **RAG框架** | LangChain | 0.1+ | 生态完善、组件丰富 |
| **向量数据库** | Chroma | 0.4+ | 轻量级、易部署 |
| **Embedding** | BGE-large-zh | - | 中文优化、开源 |
| **LLM** | DeepSeek-chat | - | 成本低、效果好 |
| **数据库** | SQLite | 3.x | 轻量级、零配置 |
| **异步框架** | asyncio | 内置 | 非阻塞 I/O |

### 3.2 技术选型理由

#### 3.2.1 为什么选择 FastAPI？

```
FastAPI 优势
├── 🚀 性能卓越：基于 Starlette，异步支持，比 Flask 快 2-3 倍
├── 📝 自动文档：Swagger UI + ReDoc，开箱即用
├── 🔒 类型安全：Pydantic 数据验证，编译时检查
├── 📚 生态丰富：与 LangChain、SQLAlchemy 等无缝集成
└── 🌐 ASGI 支持：支持 WebSocket、SSE 等现代协议
```

#### 3.2.2 为什么选择 LangChain？

```
LangChain 优势
├── 🔧 模块化设计：Document Loader、Text Splitter、Embeddings 等可插拔
├── 🔗 RAG 原生支持：create_retrieval_chain 等高级抽象
├── 💾 多数据源：支持向量库、传统数据库、API 等
├── 🧠 记忆管理：内置对话历史管理
└── 📖 文档完善：学习曲线平缓，社区活跃
```

#### 3.2.3 为什么选择 Chroma？

```
Chroma 优势
├── ⚡ 轻量级：无需独立服务，嵌入式部署
├── 💾 持久化：支持向量数据持久化存储
├── 🔄 易上手：Python 原生 API，学习成本低
├── 🤝 LangChain 集成：官方支持，一行代码集成
└── 💰 开源免费：MIT 协议，商业可用
```

#### 3.2.4 为什么选择 BGE-large-zh？

```
BGE-large-zh 优势
├── 🇨🇳 中文优化：专门针对中文语义理解优化
├── 📊 性能优异：MTEB 榜单中文排名前列
├── 🔓 开源免费：可私有部署，无需 API 费用
└── 🔢 维度适中：1024 维，兼顾精度和性能
```

### 6.3 表结构设计

#### 6.3.1 documents 表

```sql
-- 文档元数据表
CREATE TABLE documents (
    id VARCHAR(36) PRIMARY KEY,           -- UUID，文档唯一标识
    filename VARCHAR(255) NOT NULL,       -- 原始文件名
    file_path VARCHAR(500) NOT NULL,      -- 文件存储路径
    file_type VARCHAR(20) NOT NULL,       -- 文件类型：pdf/docx/txt/md
    file_size INTEGER,                     -- 文件大小（字节）
    chunk_count INTEGER DEFAULT 0,         -- 分块数量
    status VARCHAR(20) DEFAULT 'processing', -- 状态：processing/completed/failed
    description TEXT,                      -- 文档描述
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
);

-- 文档分块表（可选，用于关联向量和文档）
CREATE TABLE document_chunks (
    id VARCHAR(36) PRIMARY KEY,
    document_id VARCHAR(36) NOT NULL,
    chunk_index INTEGER NOT NULL,          -- 分块序号
    content TEXT NOT NULL,                 -- 分块内容
    vector_id VARCHAR(100),                -- Chroma 中的向量 ID
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE,
    INDEX idx_document_id (document_id)
);
```
