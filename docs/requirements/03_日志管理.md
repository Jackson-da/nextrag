# 日志管理功能需求与方案

## 1. 需求概述

| 项目 | 内容 |
|------|------|
| 功能名称 | 日志管理系统 |
| 功能描述 | 统一日志记录、请求追踪、日志轮转，支持开发/生产环境适配 |
| 优先级 | P1（重要功能） |

## 2. 需求分析

### 2.1 痛点问题

- 控制台日志分散，难以集中查看
- 生产环境无法追踪请求链路
- 日志文件无限增长，无轮转机制
- 日志格式不统一，难以解析

### 2.2 功能需求

| 需求 | 描述 |
|------|------|
| 日志分级 | 支持 DEBUG、INFO、WARNING、ERROR 级别 |
| 多输出 | 同时输出到控制台和文件 |
| 请求追踪 | 为每个请求生成 trace_id，便于追踪 |
| 日志轮转 | 文件大小超过 10MB 时自动切割，保留 7 个文件 |
| 环境适配 | 开发环境彩色输出，生产环境 JSON 格式 |

### 2.3 非功能需求

| 需求 | 描述 |
|------|------|
| 性能 | 日志记录不影响业务响应时间 |
| 可观测性 | 结构化日志便于后续接入 ELK/云日志 |
| 可用性 | 日志服务故障不影响主业务 |

## 3. 技术方案

### 3.1 技术选型

| 组件 | 选择 | 原因 |
|------|------|------|
| 日志库 | structlog | JSON 格式友好、上下文支持 |
| 文件切割 | RotatingFileHandler | 按大小切割 |
| 时间戳 | ISO 8601 | 含毫秒和时区 |

### 3.2 架构设计

```
main.py
  ├── setup_logging()      # 初始化日志
  └── LoggingMiddleware   # 请求追踪

所有模块 → get_logger() → 输出到控制台/文件
```

### 3.3 日志格式

```json
{
  "timestamp": "2026-04-27T16:00:00.000000+08:00",
  "level": "INFO",
  "trace_id": "a1b2c3d4",
  "service": "smart-qa-system",
  "logger": "app.api.chat",
  "msg": "问答完成",
  "extra": { "tokens": 512, "latency": 1.2 }
}
```

### 3.4 文件结构

```
backend/
├── app/core/logging.py     # 日志配置
├── app/middleware/logging.py # 请求中间件
└── logs/
    └── app.log             # 日志文件
```
