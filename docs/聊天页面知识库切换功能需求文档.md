# 需求文档：聊天页面知识库切换功能

**版本**：v1.0  
**编写日期**：2026-04-27  
**状态**：✅ 已实现

---

## 1. 需求背景

### 1.1 问题描述

当前系统存在以下问题：

1. **场景一**：用户A在"产品文档"知识库上传了产品介绍，用户B在"技术文档"知识库上传了API说明。当用户想同时询问两个领域的问题时，无法切换知识库。

2. **场景二**：用户从知识库列表页进入聊天页面时，系统会自动限定为该知识库的检索范围。用户无法主动切换回"全局检索"模式。

### 1.2 目标

支持在聊天页面动态切换知识库检索范围，实现：
- 切换到指定知识库进行精准检索
- 切换回全局检索模式
- 切换后新消息使用新的检索范围
- 页面顶部清晰显示当前检索模式

```

│  ┌─────────────────────────────────┐    ┌────┐  ┌─────┐   │
```

**组件说明**：
- **el-select**：下拉选择器，显示当前选中的知识库名称

- 交互直觉，符合常见UI设计模式

- 比标签页多一次点击操作

## 5. 技术实现方案

### 5.1 前端改动

**文件**：`frontend/src/views/ChatView.vue`

**改动说明**：
1. 将现有的 el-tag（显示当前知识库名称）替换为 el-select 下拉选择器
2. 添加清除按钮，提供快捷切换为全局检索的方式
3. 下拉列表包含"全部知识库"和所有知识库选项
4. 选中后触发 store 更新，调用 setKnowledgeBase 方法

**核心代码**：

```typescript
// 切换知识库
function handleKnowledgeBaseChange(kbId: string | null) {
  chatStore.setKnowledgeBase(kbId)
}

// 清除知识库（切换为全局检索）
function clearKnowledgeBase() {
  chatStore.setKnowledgeBase(null)
}
```

### 5.2 Store 改动

**文件**：`frontend/src/store/chat.ts`

**改动说明**：
1. 在 ChatState 中添加 `currentKnowledgeBaseId` 字段
2. 添加 `setKnowledgeBase` 方法
3. 发送消息时，将 knowledge_base_id 传递给 API

**核心代码**：

```typescript
// state
currentKnowledgeBaseId: string | null = null

// mutations
setKnowledgeBase(state, kbId: string | null) {
  state.currentKnowledgeBaseId = kbId
}

// actions
async sendMessage(question: string) {
  const params: any = {
    question,
    session_id: this.currentSessionId
  }
  
  // 如果选择了指定知识库，添加knowledge_base_id参数
  if (this.currentKnowledgeBaseId) {
    params.knowledge_base_id = this.currentKnowledgeBaseId
  }
  
  await this.sendChatRequest(params)
}
```

### 5.3 知识库列表获取

复用现有的 `knowledgeBaseStore` 中的 `knowledgeBaseList`，无需额外API调用。

## 6. API 接口说明

### 6.1 聊天接口

**接口**：POST /api/v1/chat/chat

**请求参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| question | string | 是 | 用户问题 |
| session_id | string | 是 | 会话ID |
| knowledge_base_id | string | 否 | 知识库ID，null或不传表示全局检索 |

**响应格式**：

```json
{
  "code": 0,
  "data": {
    "answer": "根据检索结果生成的答案",
    "references": []
  },
  "message": "success"
}
```

### 6.2 注意事项

- 后端需要支持 knowledge_base_id 为 null 时执行全局检索
- 历史消息（切换前的消息）保留原有的检索结果，不受影响
- 切换知识库后仅影响后续新消息的检索范围

## 8. 风险与约束

### 8.1 风险评估

| 风险描述 | 可能性 | 影响程度 | 缓解措施 |
|---------|--------|----------|----------|
| 后端不支持knowledge_base_id为null的全局检索 | 低 | 高 | 提前与后端确认API兼容性 |
| 切换知识库后历史消息混淆 | 低 | 中 | 明确设计：仅新消息受影响 |
| 知识库数量过多导致下拉列表过长 | 中 | 低 | 考虑添加搜索过滤功能 |

### 8.2 约束条件

- 需后端API支持 `knowledge_base_id` 参数（可为空）
- 需复用现有的知识库列表数据
- 保持向后兼容，不影响现有功能
